# Subtask v3 Design

## Problems with Current Design

1. **Workspace confusion**: Lead forgets changes are in a different directory, tries to read/merge from cwd
2. **Merge difficulty**: Lead doesn't understand worktree mechanics, copies files or pushes incorrectly
3. **Status ambiguity**: "6/6 progress + working" is confusing - worker updated progress but hasn't finished
4. **Workflow rigidity**: Default assumes implementation, but many tasks start as research

## Design Principles

1. **Leverage Claude's git skills** - Don't abstract git away, fill knowledge gaps
2. **Branch-centric, not workspace-centric** - The branch is the interface, workspace is implementation detail
3. **Workflows guide the lead** - Stages remind lead of process (plan → implement → review), preventing shortcuts
4. **Clear status** - Unambiguous whether worker is active or waiting

## Key Changes

### 1. Branch-Centric Output

After task completes, show the branch name prominently and exact git commands:

```
Task fix/auth-bug replied

Branch: fix/auth-bug

─── View changes (from project root) ───
git diff main..fix/auth-bug
git log main..fix/auth-bug --oneline

─── Merge changes ───
git merge fix/auth-bug

If there are conflicts, ask the worker to resolve them:
  subtask send fix/auth-bug "There are merge conflicts in X, please resolve"
```

**Output after worker replies (implement stage):**
```
Task fix/auth-bug replied (implement stage)
Branch: fix/auth-bug

─── View changes ───
git diff main..fix/auth-bug
git log main..fix/auth-bug --oneline

─── Merge changes ───
git merge fix/auth-bug

─── If conflicts ───
Ask worker to resolve: subtask send fix/auth-bug "conflict in X, please resolve"
```

**Why this works**: Git worktrees share history with main repo. The branch exists in both places. All these commands work from the main project directory - no need to cd anywhere.

What works from main cwd:
- `git diff main..fix/auth-bug` ✓
- `git log main..fix/auth-bug` ✓
- `git show fix/auth-bug:src/file.go` ✓
- `git merge fix/auth-bug` ✓
- `git status` ✗ (shows main worktree, not task's - but worker should commit anyway)

### 2. Status Rename

| Current | New | Meaning |
|---------|-----|---------|
| `working` | `working (3m)` | Worker is actively running |
| `replied` | `replied` | Worker finished, waiting for you |

"Paused" implies we paused it. "Replied" is clear: the worker said something, ball is in your court.

```
TASK             STATUS        PROGRESS  TITLE
fix/auth-bug     working (3m)  4/6       Fix auth token refresh
feat/metrics     replied       6/6       Add API metrics
```

**Progress display**: Only show PROGRESS when status is `replied`? Or always show but make it clear it's worker's self-report, not completion indicator.

### 3. Workflows Guide the Lead

**Key insight**: Workflows aren't just structure for workers - they're **guardrails for the lead**. Across compacts, the lead might forget to review the plan before jumping to implementation. Workflows remind them.

Keep workflows as default. They ensure the lead follows good process:
1. **Plan stage**: Research + plan together, iterate until plan is solid
2. **Implement stage**: Execute the plan

The plan stage naturally includes research. "Plan" means "figure out what to do" not just "organize steps."

```bash
# Task starts in plan stage
subtask send api-design "We need a new metrics API"

# Worker researches options, proposes plan
# Lead reviews: "What about rate limiting?"
subtask send api-design "What about rate limiting?"

# Worker updates plan
# Lead approves: move to implement stage
subtask send api-design "Plan looks good, implement it"

# Now in implement stage - worker executes
```

The workflow reminds lead at each `send`:
```
Task api-design (plan stage)

The worker proposed a plan. Review it before proceeding to implementation.
To move to implement: subtask stage api-design implement
```

**Freeform as opt-out** for simple tasks:
```bash
subtask send fix/typo --no-workflow "Fix typo in README"
```

### 4. Convenience Commands (Optional)

```bash
# Shortcut for git merge
subtask apply fix/auth-bug
# Runs: git merge fix/auth-bug

# Maybe in future: create PR instead of local merge
subtask pr fix/auth-bug
```

Not required - Claude can use git directly. Just saves typing.

## Task Lifecycle

```
                    send (new)
                       │
                       ▼
                 ┌─────────────────────┐
     ┌───────────│   working (plan)    │◄──────────┐
     │           └─────────┬───────────┘           │
     │                     │ worker finishes       │
     │                     ▼                       │
     │           ┌─────────────────────┐           │
     │           │   replied (plan)    │───send────┤  (iterate on plan)
     │           └─────────┬───────────┘           │
     │                     │ stage implement       │
     │                     ▼                       │
     │           ┌─────────────────────┐           │
     │           │ working (implement) │◄──────────┤
     │           └─────────┬───────────┘           │
     │                     │ worker finishes       │
     │                     ▼                       │
     │           ┌─────────────────────┐           │
     │           │ replied (implement) │───send────┘  (adjustments)
     │           └─────────┬───────────┘
     │                     │ merge + close
     │                     ▼
     │           ┌──────────┐
  error─────────►│  closed  │
                 └──────────┘
```

Keep `draft` status for `subtask draft` command (prepare task without running).

## Decisions

1. **Workspace path**: Only show in `subtask show <task>` for debugging. Branch name is the primary interface.

2. **Progress while working**: Show with `...` indicator: `4/6...` to make clear it's partial/in-progress.

3. **Questions**: Use `subtask ask` command (no task, runs in cwd, memorable session names).

4. **PROGRESS.json**: Optional. Workers can create it with or without workflows. We show it when it exists.

## Example Session (v3)

```bash
# Start a task (enters plan stage)
$ subtask send fix/auth "The auth token refresh is failing intermittently"

Task fix/auth started (plan stage)
Branch: fix/auth

Worker is researching the issue and will propose a plan.

# Check status while working
$ subtask list

TASK       STATUS        PROGRESS  TITLE
fix/auth   working (2m)  1/3...    The auth token refresh...

# Worker finishes plan stage
$ subtask list

TASK       STATUS    PROGRESS  TITLE
fix/auth   replied   2/3       The auth token refresh...

# Review the plan
$ subtask show fix/auth
# Shows conversation, plan details...

# Approve and proceed to implementation
$ subtask send fix/auth "Plan looks good. Implement it."

Task fix/auth resumed (implement stage)
Branch: fix/auth

Worker is implementing the plan.

# Later, worker finishes
$ subtask list

TASK       STATUS    PROGRESS  TITLE
fix/auth   replied   3/3       The auth token refresh...

# See what changed (from project root - no cd needed)
$ git diff main..fix/auth
$ git log main..fix/auth --oneline

# Merge it
$ git merge fix/auth

# Done
$ subtask close fix/auth
```

**If there are merge conflicts:**
```bash
$ git merge fix/auth
# CONFLICT in src/auth.go

# Ask worker to resolve (they have full context)
$ subtask send fix/auth "Merge conflict in src/auth.go with recent changes to token validation. Please resolve and commit."

# Worker resolves in their workspace, commits, replies

# Now merge cleanly
$ git merge fix/auth
$ subtask close fix/auth
```

**For simple tasks (skip workflow):**
```bash
$ subtask send fix/typo --no-workflow "Fix typo in README: 'recieve' -> 'receive'"

# No stages, just does it and replies
```
